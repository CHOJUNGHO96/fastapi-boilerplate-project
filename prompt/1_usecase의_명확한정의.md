너는 Clean Architecture / DDD / Application Layer 설계에 매우 능숙한
시니어 백엔드 아키텍트다.

내 목표는 "Use Case"의 역할과 경계를 **개념적으로도, 실무적으로도**
절대 헷갈리지 않게 정의하는 것이다.

아래 조건을 반드시 지켜서 답변하라.

[답변 목표]
1. Use Case의 단일 책임을 한 문장으로 정의할 것
2. Endpoint / Service / Domain Service 와의 차이를 명확히 구분할 것
3. “이 로직이 Use Case인지 아닌지” 판단할 수 있는 기준을 제시할 것
4. 실무에서 흔히 발생하는 잘못된 설계 예시와 올바른 예시를 함께 제시할 것
5. FastAPI / Spring / 일반적인 백엔드 구조 어디에 적용해도 통하는 정의일 것

[설명 방식 제약]
- 추상적인 철학 설명만 하지 말 것
- 반드시 코드 구조 관점에서 설명할 것
- ‘왜 존재하는가’ → ‘무엇을 책임지는가’ → ‘무엇을 절대 하면 안 되는가’ 순서로 설명할 것
- 용어는 Application Layer 관점에서 통일할 것

[반드시 포함할 항목]
- Use Case의 정의 (One-liner)
- Use Case가 다루는 것 / 다루지 않는 것
- Service와의 경계 기준
- 트랜잭션과 Use Case의 관계
- “Use Case를 만들어야 하는 신호” 체크리스트
- 초보자가 가장 많이 하는 오해 Top 3

[금지 사항]
- Controller에 비즈니스 로직을 두는 것을 정당화하지 말 것
- Service를 만능 계층처럼 설명하지 말 것
- “팀마다 다르다” 같은 회피성 답변 금지

최종 결과는
“이 문서를 읽고 나면 Use Case를 안 쓸 수 없게 되는 수준”으로 작성하라.
